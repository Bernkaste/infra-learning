# Incident Postmortem Guide（書き方・運用ルール）

## 目的

- 同じ障害を繰り返さない（再発防止）
- 障害対応を速くする（検知/切り分け/復旧の改善）
- 属人性を減らす（知識をチーム資産にする）

## 大原則（Blameless）

- 個人を責めない（責めても再発は減らない）
- 責めるべきは「仕組み・設計・プロセスの穴」

## 書き方のコツ

### 1) まずは「影響」と「タイムライン」

- ここは推測禁止。事実だけ。
- 後から原因を考えても、事実が揺れると全部崩れる。

### 2) 「直接原因」と「根本原因」は別物

- 直接原因：最後に壊したもの（例：設定値ミス）
- 根本原因：それが本番に入った理由（例：レビュー観点不足、検証環境不備、監視不足）

### 3) アクションは “仕組み” を変える

悪い例：

- もっと気をつける
- 次から注意する
- 手順を共有する

良い例：

- CIで設定検証を自動化する（テストを追加）
- 監視にSLOアラートを追加する（検知を自動化）
- 変更管理にチェックリストを追加する（漏れを仕組みで潰す）

### 4) アクションには必ず「成果物」を持たせる

- PR URL / ドキュメント / ダッシュボード / 設定変更
- 「やったかどうか」が後から見える状態にする

## 運用ルール（おすすめ）

- 復旧後24時間以内に “サマリー” と “タイムライン” まで埋める
- 3営業日以内にレビューしてクローズ判断
- アクションはすべてチケット化して追跡する（ポストモーテムにチェックボックスだけ残さない）

## レビュー時のチェック観点

- 影響は定量で書けているか？
- タイムラインは事実のみか？
- 根本原因が「仕組み」に落ちているか？
- アクションが曖昧でないか（担当/期限/成果物あり）？
