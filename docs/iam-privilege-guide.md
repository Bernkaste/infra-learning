## IAM最小権限設計ガイド

### このドキュメントの目的

- AWSの権限（IAM）を「最小権限」で設計・運用するための考え方を、初見でも判断できる形で整理する
- 組織運用を想定し、長期キー回避 / Role前提 / 権限分離を満たす標準を定義する

---

## 1. まず押さえる全体像

1. 人はSSOでログインし、Role（作業用の権限）で作業する
2. 長期キー（Access Key）は原則禁止。必要な場合は例外として期限付きで扱う
3. 「閲覧」と「変更」、「運用作業」と「管理者作業」、「人の操作」と「自動処理」を分離し、事故の影響範囲を最小化し、監査性を最大化する

---

## 2. 設計原則（最小権限の判断基準）

### 2.1 最小権限とは

- 「必要な操作だけを許可し、それ以外は許可しない」こと
- 許可は次の3要素を絞るのが基本
    - Action：何の操作を許可するか（例：S3を一覧表示、EC2を起動など）
    - Resource：どの対象に対して許可するか（例：特定のバケット、特定のKMSキー）
    - Condition：いつ/どんな条件なら許可するか（例：MFA必須、特定リージョンのみ、タグ一致）

### 2.2 目標（満たすべき状態）

- 長期キーを避ける：漏洩しても被害が長期化しない
- Role前提：個人アカウント（IAM User）での権限付与を避け、役割単位に統一する
- 権限分離：強い権限を常用しない。人と自動化を混ぜない

---

## 3. 用語集（初見でも分かる定義 + 使いどころ）

以下は、権限設計を「誰が」「どうやってAWSに入り」「どの権限で」「何をできるようにするか」に分解した部品。

### 3.1 IAM（Identity and Access Management）

- 何：AWSの「認証（ログイン）」と「認可（権限）」を管理する仕組み
- 使いどころ：権限設計の中心

### 3.2 IAM User（IAMユーザー）

- 何：AWSアカウント内に作る“ユーザー”。パスワードやAccess Keyを持てる
- 推奨：人間の運用では原則使わない（SSO/Roleへ寄せる）
- 典型ユースケース（例外）：Access Keyしか使えない外部ツール連携、移行期の暫定
- 長所：仕組みが単純で導入が容易（小規模・一時的用途では手早い）。既存システム/外部ツールで"アクセスキー前提"のものに対応しやすい
- できない/弱点：長期キー運用になりやすい。漏洩時の被害が大きい。棚卸し/無効化が面倒になりがち

### 3.3 Access Key（アクセスキー、長期キー）

- 何：AWS CLI/SDKが使う“IDとパスワード”のセット（Access Key ID / Secret Access Key）
- 推奨：原則禁止
- 例外時の必須条件：期限、最小権限、秘匿保管（平文禁止）、記録（理由・責任者・削除日）
- 長所：多くのツールが対応
- できない/弱点：漏れたら消すまで使われ続ける（長期化）。誤コミット等の事故が多い

### 3.4 IAM Role（ロール）

- 何：権限の“箱（作業服）”。ユーザーではなく、権限のまとまり
- 使いどころ：組織運用の標準。人の操作・自動処理・AWSサービスはRoleで権限を持つ
- 長所：長期キーを持たせずに運用しやすい。用途別に分離できる
- できない/注意：Roleを安全にするにはTrust Policy（誰が使えるか）を絞る必要がある

### 3.5 AssumeRole（ロールを引き受ける） / STS（Security Token Service / 短期クレデンシャル）

- 何：主体（人/機械/サービス）がRoleに“着替える”操作。結果としてSTSが期限付きの鍵（短期資格情報）を発行する
- 重要：AssumeRoleは「ログイン」ではなく「権限の切替」
- 使いどころ
    - 人が作業用Roleに切り替える（SSOでも裏側で起きている）
    - Terraform用Roleへ切替してapplyする（権限分離）
    - クロスアカウントで別アカウントのRoleを使う（監査アカウント→本番など）
    - GitHub Actions等がRoleを引き受ける（OIDC）
- 長所：短期で期限切れになるため漏洩の影響が限定される。誰がどのRoleを使ったか追える
- できない/注意
    - 権限を勝手に増やすものではない（Roleに許可が無ければ何もできない）
    - Trust Policyがザルだと誰でも強権限Roleを使える

### 3.6 SSO（IAM Identity Center）

- 何：組織の“入館ゲート”。個人IAM Userを作らずにログイン管理を統一する仕組み
- 使いどころ：組織運用の第一選択肢
- 長所：入退社・異動の権限管理が容易。MFAやポリシーを統一できる
- できない/注意：サービス間権限（EC2→S3等）は別途サービスロールで設計する

### 3.7 Permission Set（SSOの権限パック）

- 何：SSOで割り当てる権限テンプレ。AWSアカウント側ではRoleとして実体化して使われる
- 使いどころ：ReadOnly/Operator/Adminなどの標準ロール配布
- 長所：権限配布が標準化される
- できない/注意：Permission Set自体が“万能”ではなく、最小権限の中身はポリシー設計次第

### 3.8 Policy（権限ポリシー） / Trust Policy（信頼ポリシー）

- Permissions Policy（権限ポリシー）
    - 何：そのRoleで「何ができるか」
    - 例：S3の読み取りのみ許可
- Trust Policy（信頼ポリシー）
    - 何：そのRoleを「誰がAssumeできるか」
    - 例：SSOユーザーだけ、特定のCIだけ
- 長所：誰が使えるか（Trust）と、何ができるか（Permissions）を分離できる
- 注意：Trustだけ整えても権限は増えない。Permissionsだけ強いとTrustが穴になる

### 3.9 OIDC（OpenID Connect）

- 何：GitHub Actions等が「自分はこのワークフローです」を証明してAWS Roleを引き受ける仕組み
- 使いどころ：CI/CDでAccess Keyを使わずにAWSへアクセス
- 長所：長期キーをCIに置かなくてよい
- 注意：Trust Policyで対象リポジトリ/ブランチ等を絞らないと危険

### 3.10 Permission Boundary（権限の上限）

- 何：IAM User/Roleが「どれだけ強い権限を持てるか」の天井
- 使いどころ：現場に権限管理を委任したいが、上限は超えさせたくない場合
- 長所：委任と統制の両立
- できない/注意：Boundaryは許可を与えない（上限を作るだけ）

### 3.11 SCP（Service Control Policy）

- 何：Organizationsで組織全体に効く“絶対禁止”ルール
- 使いどころ：全アカウント共通の禁止（例：特定リージョン禁止、CloudTrail停止禁止）
- 長所：組織全体で強制できる
- できない/注意：SCPは許可を与えない（上限を削るだけ）

### 3.12 CloudTrail（監査ログ）

- 何：AWS操作の記録（誰がいつ何をしたか）
- 使いどころ：事故調査・監査の前提
- 注意：ログ無効化/削除権限は強権限なので分離・制限対象になりやすい

## 4. どちらを使う？判断ガイド

### 4.1 IAM User（アクセスキー） vs SSO（Role）

- 原則：SSO + Role
- IAM User（アクセスキー）を使うのは「例外」
    - 外部ツールの制約など、Access Keyが必須の場合のみ
    - 例外時は期限・最小権限・秘匿保管・記録が必須

### 4.2 Role vs AssumeRole（STS）

- Role：権限の“定義（作業服）”
- AssumeRole：権限の“切替（着替え）”
- 使い分け
    - 普段はReadOnly/Operatorで作業
    - applyや管理作業など高リスク操作だけ、期限付きで強いRoleに切り替える（Just-In-Time）

### 4.3 Permissions Policy vs Trust Policy

- Permissions：何ができるか（操作内容の最小化）
- Trust：誰が使えるか（主体の最小化）
- 原則：両方を絞って初めて安全

### 4.4 Permission Boundary vs SCP

- Boundary：アカウント内での上限（委任管理に強い）
- SCP：組織全体の絶対禁止（全体統制に強い）
- どちらも「許可を増やさない」。上限を削る仕組み

## 5. 標準ユースケース（設計例のテンプレ）

### 5.1 人間の作業（組織標準）

- 方式：SSOログイン → Permission Set（ReadOnly/Operator） → 作業
- apply等の変更は、必要ならTerraformDeploy Roleへ切替（期限付き）
- 目的：長期キー不要、監査可能、権限分離

### 5.2 WSL上でAWS CLI + Terraformを実行したい（個人PC）

- 推奨：WSL内のAWS CLI v2でSSOログインし、短期資格情報を使ってTerraformを実行
- メリット：Access Keyをローカル保存しない運用に移行できる
- 注意：WSLとWindowsで`~/.aws`が別になりやすい（認証情報の置き場所を混ぜない）
→WSLとWindowsのどちらのターミナルで実行するのか、どちらの設定ファイル/認証キャッシュを使用するのかを一致させる

### 5.3 CI/CD（GitHub Actions等）

- 推奨：OIDC → CI用RoleをAssumeRole（短期）
- 目的：CIにAccess Keyを置かない
- 注意：Trust Policyでリポジトリ/ブランチ/環境を絞る

### 5.4 クロスアカウント（監査/運用分離）

- 方式：監査アカウントのRole → 本番アカウントのReadOnly RoleをAssume
- 目的：権限分離、事故時の影響範囲を分割

---

## 6. ありがちな落とし穴

- IAM User（個人）に強権限（Administrator相当）を直付けして常用している
- Access Keyがどこかに平文で残っている（Slack/リポジトリ/端末）
- Trust Policyが広すぎて、誰でも強権限RoleをAssumeできる
- ReadOnlyと変更権限が分離されていない（調査でも壊せる）
- 監査ログ（CloudTrail）を無効化できる権限が広く配られている
